# SWIG C++ to Python Example

This project demonstrates how to expose a small C++ class to Python using SWIG. You will generate wrapper code, build a native Python extension, and import it from regular Python code.

Key names used in this repo:
- C++ source: `calculator.cpp`
- SWIG interface file: `cpp_to_python.i`
- Python module: `rechner.py`

## Overview

A native Python extension is a compiled binary module that Python can import. On Windows, the extension file is a `.pyd`. It contains machine code produced by a C++ compiler plus the SWIG-generated wrapper. Python loads it the same way it loads a normal module, but the code runs at native speed.

SWIG (Simplified Wrapper and Interface Generator) reads the interface file and generates wrapper code. That wrapper is compiled into the `.pyd`, and Python calls into C++ through it.

## Project Files

- `calculator.cpp`
  - The C++ implementation. Defines a `Calculator` class with math methods.

- `cpp_to_python.i`
  - The SWIG interface file. Declares the module name and what to expose.

- `cpp_to_python_wrap.cxx`
  - Generated by SWIG. Do not edit directly.

- `rechner.py`
  - Generated by SWIG. Pure Python proxy that imports `_rechner` and exposes
    a friendly `Calculator` class.

- `setup.py`
  - Build script that compiles the wrapper and C++ source into a Python
    extension.

- `test_math.py`
  - Simple test script that imports the module and calls its methods.

Note: On Windows, the compiled extension includes an ABI/CPython tag in the filename (PEP 3149). For example `_rechner.cp39-win_amd64.pyd` is still the `_rechner` module; Python resolves `import _rechner` to the tagged filename.

## Build Instructions

### 1) Generate the SWIG wrapper

```
swig -c++ -python cpp_to_python.i
```

Outputs:
- `cpp_to_python_wrap.cxx`
- `rechner.py`

### 2) Build the Python extension

```
py -3 .\setup.py build_ext --inplace
```

Outputs:
- `_rechner.pyd` (or a tagged variant like `_rechner.cp39-win_amd64.pyd`)

### 3) Run the test

```
py -3 .\test_math.py
```

Expected output is similar to:

```
Add: 5 + 3 = 8.0
Multiply: 7 * 4 = 28.0
Power: 2^10 = 1024.0
Square Root: 16 = 4.0
```

## How the Import Works

1) You run `import rechner`.
2) Python loads `rechner.py` (generated by SWIG).
3) `rechner.py` imports `_rechner`.
4) Python loads the compiled `_rechner.pyd` extension.

In short, the `.pyd` is the actual C++ extension that Python loads.

## Why the Interface File Looks the Way It Does

`cpp_to_python.i` has three key parts:

1) `%module rechner`
   - Sets the Python module name you import.

2) The `%{ ... %}` block
   - Copied directly into the generated C++ wrapper.
   - Must include the real C++ header or source so the wrapper can compile.

3) The class declaration
   - Lists the C++ class and methods to expose.
   - SWIG uses this to generate the Python bindings.

## What `setup.py` Does

`setup.py` tells Python how to build a native extension module:

- Points to the source files to compile.
- Marks the extension as C++.
- Locates Python headers and libraries for linking.
- Produces `_rechner.pyd` (the actual C++ extension module).

Without `setup.py`, Python would not know how to compile the wrapper into a module that `import` can load.

## Common Mistakes

- Editing `cpp_to_python_wrap.cxx` or `rechner.py` directly.
  - These are generated files. Edit `cpp_to_python.i` instead, then re-run SWIG.

- Forgetting to rebuild after C++ changes.
  - Re-run SWIG and then re-run `setup.py`.
